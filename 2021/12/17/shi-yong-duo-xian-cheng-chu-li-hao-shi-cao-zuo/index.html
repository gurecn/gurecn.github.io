使用多线程处理耗时操作
优化
Android
发布日期:   2021-12-17
作者:   Gure
文章字数:   1.7k
阅读时长:   6 分
阅读次数:  

需求背景

Android开发中，当UI线程被阻塞会导致ANR的出现。
尽管开发原则应该是主要逻辑都交给服务器进行处理，APP端重点进行界面的优化与数据的显示。但有些问题仍需要APP端进行大量的逻辑运算。这时很容易出现程序未响应的情况或者列表滚动卡顿、刷新延迟等问题。
鉴于异步处理与手机处理性能等多种原因的考虑，使用多线程来处理该问题是个有效的解决方式。
在实际开发中，多线程使用非常广泛。
我们应用中消息模块就遇到了服务器无法处理的逻辑问题：我们项目消息模块是使用环信平台，好友体系是自己实现。这导致一个问题：新来的消息发送者是好友、关注、粉丝或陌生人呢？
产品设计如下：如果是好友、关注、已聊过的消息，那么直接显示在好友列表；如果是其他人的消息，那么仅在消息列表的陌生人消息处显示消息的数量；如果是对群操作的透传消息，那么仅在消息提醒条目上显示提醒的数量；如果是公众号发来的消息，那么仅在公众消息条目上显示消息数量。同时没有消息的条目是不显示的，在聊天界面可以清空消息和删除某一条消息，当把本人发送的消息全部删除后，则聊天状态变为未聊过。
在这个前提下，逻辑是相对复杂的：
在初次布局时，需遍历一遍消息，整体进行布局显示。每来一条消息，首先获取到发送人，区分透传或者普通消息。若是普通消息，则区分是群消息或者个人消息。若是个人消息需要获取到发送者，然后获取关系和聊天情况……等等一系列逻辑判断。
刚开始是直接在主线程处理，发现初次显示需要等待3S左右。并且每次来新消息都会刷新一下消息列表（ListView），由于ListView不能部分刷新，可以明显感觉到刷新卡顿。并且当来的消息很多的时候，由于ListView需要不断布局，导致点击失效的情况。

结果过程

其实在这种情况下使用多线程来组织数据，然后使用Handler进行界面的刷新是一个非常好的选择。
创建子线程进行数据组织的代码很简单，直接贴出关键部分，不再赘述。

LinkedList tempMsgList = new LinkedList<>();//存放消息的集合
FreshMsg mr = new FreshMsg();
Thread t = new Thread(mr);
t.start();
/**
 * 创建子线程进行消息的刷新及消息界面的重新布局
 */
class FreshMsg implements Runnable {
    @Override
    public void run() {
        //消息组织代码，将组织好的消息存放到tempMsgList中
        handler.sendEmptyMessage(0);
    }
}
@SuppressLint("HandlerLeak") 
Handler handler = new Handler() {
    //Handler负责将组织好的消息列表显示
    @Override
    public void handleMessage(android.os.Message message) {
        if(messageListAdapter!=null){
            msgList.clear();
            msgList.addAll(tempMsgList);
            messageListAdapter.notifyDataSetChanged();
        }
    }
};

大概看一下，发现该代码不存在问题。经测试发现，卡顿的问题解决，不再出现条目无法点击的情况。
但是会概率出现条目重复的问题。
这是什么原因呢？考虑整个处理逻辑发现，产生重复肯定是tempMsgList中存放了重复的消息，为什么会重复呢？
当消息来的非常频繁的时候，当一条消息到来立即开始新的子线程任务（定义成任务1）。新的消息再次到来时，tempMsgList被重新创建，之前的数据被清空，然后会重新创建新的子线程来处理消息（定义成任务2）。请注意，这时任务1和任务2均没有结束，均会操作tempMsgList，任务1会将剩下的消息继续处理并把得到的结果存放到tempMsgList；同时的任务2也会进行同样的操作，导致消息重复显示。
问题找到了，那么就解决吧。对于这种多线程异步处理的情况解决方式可以加锁：在任务1未结束的时候不让任务2执行。修改后的代码就是如下：

/**
 * 创建子线程进行消息的刷新及消息界面的重新布局
 */
class FreshMsg implements Runnable {
    @Override
    public void run() {
        synchronized (this) {     //消息频繁时，会出现异步实现问题，加锁控制
            tempMsgList = new LinkedList<>();//存放消息的集合
            //消息组织代码，将组织好的消息存放到tempMsgList中
            handler.sendEmptyMessage(0);
        }
    }
}

加锁了问题应该被解决了。
但测试发现为题依旧。为什么呢？自己明明加锁了，怎么还能出现重复的情况啊。
当时自己觉得很可不思议。仔细看代码，终于发现了问题所在：这段代码锁是加上了，也确实约束了线程的执行。现在的情况下是任务2会一直等待任务1执行完，然后才进入所内部进行执行。但是执行顺序正确了，结果还是没有改变啊。
原来tempMsgList是引用数据，在创建任务2时tempMsgList已经被重置了。这时任务1执行后的数据保存的位置是新的tempMsgList集合，任务2再次执行时会继续向tempMsgList添加数据而不会重新清空，这必然导致数据的重复。
其实这篇文章重点想告诫自己，在多线程中处理引用型数据时要注意引用的地址是不是当前的地址，会不会在线程执行时地址遭到改变而影响以后的结果。
原因找到，解决就很简单：只需在创建线程时不重置集合，在子线程内部将集合重置即可：

/**
 * 创建子线程进行消息的刷新及消息界面的重新布局
 */
class FreshMsg implements Runnable {
    @Override
    public void run() {
        synchronized (this) {     //消息频繁时，会出现异步实现问题，特加锁实现
            tempMsgList = new LinkedList<>();//存放消息的集合
            //消息组织代码，将组织好的消息存放到tempMsgList中
            handler.sendEmptyMessage(0);
        }
    }
}

总结

本身是一个很小的问题，却让我印象深刻，在微小中彰显细致。这样来看卡顿和点击失效就解决了。
但ListView显示效率低下，每次刷新时，尽管数据已准备好，但是头像图片仍然会闪烁。这就需要修改ListView的控件来实现。在我将其替换成RecyclerView后，问题就不存在了。
当然项目还是有很多需优化的点。比如，来多条新消息时不需多次刷新，仅最后一条刷新，这样能很大的节省手机的性能。
