Android Home键监听方式探讨
应用
Android
发布日期:   2021-10-26
作者:   Gure
文章字数:   1.2k
阅读时长:   4 分
阅读次数:  

遇到问题

由于我公司产品要实现调起键盘界面时能直接弹出对话框，让用户直接进行输入交互。Android中，在非界面类控件下想要直接弹出对话框需要使用覆盖其他应用的权限。该需求我们使用Dialog，采用设置TYPE_SYSTEM_ALERT权限来实现：

Window window = mDialog.getWindow();
WindowManager.LayoutParams wl = window.getAttributes();
wl.type = WindowManager.LayoutParams.TYPE_SYSTEM_ALERT;
mDialog.onWindowAttributesChanged(wl);

实际使用中发现按键交互存在问题。在该对话框显示时，点击返回键会隐藏对话框，再次点击才会执行返回效果，这正是我们需要的。

但是当点击HOME键呢，会出现异常情况：部分手机上对话框没有隐藏，手机却切换到了桌面（应用隐藏），即系统返回到了主界面，但是对话框仍然在覆盖在桌面上，这显然是不合理的。今天来探究一下对话框和HOME键冲突的问题。

解决思路

Android手机按键方案一直在变化，按键的监听是个很头疼的问题。系统禁止用户重写HOME键的监听方法，就导致部分手机可能出现各类适配性问题。

解决方案首先想到的就是拦应用截点击HOME键监听点击操作。在应用监听到点击操作后，首先执行隐藏对话框的操作，然后再次点击时返回到桌面。那么技术问题就出现了：如何监听HOME键的点击呢？

Android系统对HOME键的点击确实有发送对应的广播，即：Intent.ACTION_CLOSE_SYSTEM_DIALOGS。那么实现对该广播的监听是否就能解决问题？

监听广播有几种方案，我们逐方案进行验证。

第一种方式：静态方式注册监听：

首先，在清单文件中注册广播接收者，接收系统广播：

<receiver
   android:name=".receiver.HomeKeyEventReceiver"
   android:enabled="true"
   android:exported="false" />

然后实现广播接收者主要是监听两种动作：homekey和recentapps，发现点击HOME调用的是homekey动作，长按HOME键是调用了recentapps动作，故实现监听这两种动作即可：

public class HomeKeyEventReceiver extends BroadcastReceiver {
    String SYSTEM_REASON = "reason";
    String SYSTEM_HOME_KEY = "homekey";
    String SYSTEM_HOME_KEY_LONG = "recentapps";
    public HomeKeyEventReceiver() {
    }

    @Override
    public void onReceive(Context context, Intent intent) {
        String action = intent.getAction();
        if (Intent.ACTION_CLOSE_SYSTEM_DIALOGS.equals(action)) {
            String reason = intent.getStringExtra(SYSTEM_REASON);
            if (TextUtils.equals(reason, SYSTEM_HOME_KEY) || TextUtils.equals(reason, SYSTEM_HOME_KEY_LONG)) {
                //点击HOME键
                LauncherModel.getInstance().getGlobalEventBus().post(new OnHomeClickEvent());
            }
        }
    }
}

然后使用红米手机测试程序效果：点击HOME键，查看效果，期待日志的产生。

结果让人失望，竟然没有如愿的实现监听！！！

为什么呢？是小米手机做了特殊处理还是Android系统做了系统级别的限制，这个不可而知了。

既然静态不行，那就动态试试。

第二种方式：使用动态方式注册广播接收者，监听HOME键的点击事件。

为了实现类似静态的效果，在整个程序内监听HOME键的点击，所以我们在Application中进行动态广播的注册，代码如下：

private void startServerAndReceiver() {
    IntentFilter filterHome = new IntentFilter(Intent.ACTION_CLOSE_SYSTEM_DIALOGS);
    getAppContext().registerReceiver(new HomeKeyEventReceiver(), filterHome);
}

发现，终于，日志出现了！是时候尽情庆祝吧。

不过，等等。好像这种方式也有漏洞。。

经测试组测试发现，在部分国产定制ROM中，发现一个奇怪的现象：在部分手机上使用动态监听依然无法达到监听HOME键的目的，比如说在奇酷手机。这又是为什么呢？

经过研究发现，注册动态广播需要在主线程进行，我们对代码进行改进，终于实现了对奇酷的监听。

@Override
public void onCreate() {
    super.onCreate();
    IntentFilter filterHome = new IntentFilter(Intent.ACTION_CLOSE_SYSTEM_DIALOGS);
    getAppContext().registerReceiver(new HomeKeyEventReceiver(), filterHome);

}

由于手机资源有限，不能对所有手机进行测试，希望大家有更好的方式或者想法能够及时反馈。

总结

  1. HOME键的点击操作在清单文件中静态注册无法达到预期效果。可能是系统限制了第三方应用对该广播的静态监听，也可能是广播优先级问题。

  2. 在动态注册广播时，为了实现静态注册的效果，可以在Application中注册。在Application中注册需要注意要在主线程中注册，使用Application的Context。Application为保证应用启动速度，经常在子线程中进行初始化操作，广播监听需注意：要在主线程中进行。

  3. 针对国产厂商定制的ROM需要重点关注。部分厂商不再遵循Google规范，取消广播、拦截广播的情况经常出现。在做机型适配的时候最好能够尽量全面的测试不同机型。
