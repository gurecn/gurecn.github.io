Android Studio编译Jar包运行异常
混淆
Java
发布日期:   2022-01-24
作者:   Gure
文章字数:   1.8k
阅读时长:   7 分
阅读次数:  

上面两篇博客介绍了Jar包如何进行签名和反编译，今天介绍一下使用Android studio编译Jar包及使用的问题。
首先介绍一下如何使用Android studio编译Jar包。

编译Jar文件

首先我们通过Android studio打开需要编译成Jar包的项目，然后找到build.gradle文件，创建一个task，命名成makeJar，内容如下：

task makeJar  {
    copy {
        from('build/intermediates/bundles/release/')
        into('build/libs/')
        include('classes.jar')
        rename('classes.jar', "uSDK_Java.jar".toLowerCase())
    }
}

makeJar.dependsOn(build)

此时，重新编译项目，会发现在右侧Gradle菜单栏内，依次点击app-Tasks-other-makeJar。双击makeJar，会进行编译操作。当编译成功后，会输出如下日志(截取关键部分)：

14:42:30: Executing task ‘makeJar’…
Executing tasks: [makeJar] in project C:\develop\DevProject\Android\Library\app
:app:preBuild UP-TO-DATE
:app:preDebugUnitTestBuild UP-TO-DATE
:app:javaPreCompileDebugUnitTest
:app:compileDebugUnitTestJavaWithJavac NO-SOURCE
:app:mockableAndroidJar
:app:test UP-TO-DATE
:app:check
:app:build
:app:makeJar
Deprecated Gradle features were used in this build, making it incompatible with Gradle 5.0.
See https://docs.gradle.org/4.6/userguide/command_line_interface.html#sec:command_line_warnings
BUILD SUCCESSFUL in 28s
55 actionable tasks: 29 executed, 26 up-to-date
14:43:00: Task execution finished ‘makeJar’.

我们看到BUILD SUCCESSFUL提示，因为Task中我们设置目标文件输出到build/libs/，我们打开项目中app/build/libs路径，会看到输出的Jar包（若是没有创建libs文件夹，可以再执行一次makeJar，在我这边发现需要执行两次才能输出）。
此时我们已经能够将Java文件打包成Jar文件了。

混淆配置

此时我们解压Jar包，可以发现资源文件和方法都没有混淆，下面我们开启混淆功能。
我们继续找到build.gradle文件，配置buildTypes方法:

buildTypes {
    release {
        minifyEnabled true
        proguardFiles getDefaultProguardFile('proguard-android.txt'), 'proguard-rules.pro'
    }
    debug {
        minifyEnabled true
        proguardFiles getDefaultProguardFile('proguard-android.txt'), 'proguard-rules.pro'
    }
}

然后在app路径下创建proguard-rules.pro文件，在文件内配置混淆规则。

# 代码混淆压缩比，在0~7之间，默认为5，一般不做修改
-optimizationpasses 5
# 混合时不使用大小写混合，混合后的类名为小写
-dontusemixedcaseclassnames
# 指定不去忽略非公共库的类
-dontskipnonpubliclibraryclasses
# 这句话能够使我们的项目混淆后产生映射文件
# 包含有类名->混淆后类名的映射关系
-verbose
# 指定不去忽略非公共库的类成员
-dontskipnonpubliclibraryclassmembers
# 不做预校验，preverify是proguard的四个步骤之一，Android不需要preverify，去掉这一步能够加快混淆速度。
-dontpreverify
# 保留Annotation不混淆
-keepattributes *Annotation*,InnerClasses
# 避免混淆泛型
-keepattributes Signature
# 抛出异常时保留代码行号
-keepattributes SourceFile,LineNumberTable
# 指定混淆是采用的算法，后面的参数是一个过滤器
# 这个过滤器是谷歌推荐的算法，一般不做更改
-optimizations !code/simplification/cast,!field/*,!class/merging/*
# 保留本地native方法不被混淆
-keepclasseswithmembernames class * {
    native ;
}
# 保留bitcoin方法不被混淆
-keep class org.bitcoin.** {*;}
-keep class org.bitcoinj.** {*;}
#所有View的子类及其子类的get、set方法都不进行混淆
-keepclassmembers public class * extends android.view.View {
   void set*(***);
   *** get*();
}
# 不混淆Activity中参数类型为View的所有方法
-keepclassmembers class * extends android.app.Activity {
   public void *(android.view.View);
}
# 不混淆Enum类型的指定方法
-keepclassmembers enum * {
    public static **[] values();
    public static ** valueOf(java.lang.String);
}
# 不混淆Parcelable和它的子类，还有Creator成员变量
-keepclassmembers class * implements android.os.Parcelable {
  public static final android.os.Parcelable$Creator CREATOR;
}

# 不提示兼容库的错误警告
-dontwarn android.support.**
#忽略警告
-ignorewarnings
#保护泛型
-keepattributes InnerClasses,Signature

需注意，项目中有一部分对外调用的方法和Java文件,或者引用的第三方资源是不能混淆的，否则会在运行时找不到对应的Java类或方法。
此时我们再次运行makeJar，重新编译Jar文件，发现输出的Jar文件被混淆了。

解决问题

此时我们将Jar文件放到Android项目进行依赖使用是没问题的，各项功能均正常。但是当我们把Jar文件依赖到Java工程或Spring项目时，就会报错：

Exception in thread “main” java.lang.VerifyError: Expecting a stackmap frame at branch target 18
Exception Details:
Location:
com/test/encrypt/sdk.nSdkEncrypt([B)[B @11: ifne
Reason:
Expected stackmap frame at this location.
Bytecode:
0x0000000: 1205 120b b800 372a b700 2b9a 0007 03bc
0x0000010: 08b0 2bb8 0031 b0
at springboottest.springboot.TestSdkService.init(TestSdkService.java:14)
at springboottest.springboot.TestSdkService.main(TestSdkService.java:60)

看错误代码是一个VerifyError错误。
网上查询资料，有相关解决方案，比如：

在项目的tool bar中，看到自己的项目，点击三角选择：“Edit configuration”
把VM option改成“-noverify”

此时运行发现正常了。
但是这种方法治标不治本，发现Spring项目配置该参数无法生效。并且配置noverify存在很大的安全风险，项目安全扫描有可能会报风险，所以最终无法使用该取巧的方案解决，还需要从编译Jar方案上找原因并解决。
测试几次，我发现，在混淆配置中，只要关闭混淆，那么编译的包就没有问题，只要开启混淆，哪怕项目中所欲代码都通过keep配置成不混淆，那么编译的包也不行。可用的配置如下：

buildTypes {
    release {
        minifyEnabled false
        proguardFiles getDefaultProguardFile('proguard-android.txt'), 'proguard-rules.pro'
    }
    debug {
        minifyEnabled false
        proguardFiles getDefaultProguardFile('proguard-android.txt'), 'proguard-rules.pro'
    }
}

这样看了，确实是混淆出现了问题且不是代码本身的问题。
经过查看混淆配置，发现有一个配置项：

#不做预校验，preverify是proguard的四个步骤之一，Android不需要preverify，去掉这一步能够加快混淆速度。
-dontpreverify

原来Android在使用Jar等资源时，为了节约时间，去掉了预校验功能。所以我们编译的Jar文件在Android项目使用没有问题，但是在Java 项目中会报错。我们关闭该策略，再次编译，发现仍然不行。再次分析发现，原来在build.gradle中，我们引用了系统预置的混淆配置文件proguard-android.txt：

buildTypes {
    release {
        minifyEnabled false
        proguardFiles getDefaultProguardFile('proguard-android.txt'), 'proguard-rules.pro'
    }
    debug {
        minifyEnabled false
        proguardFiles getDefaultProguardFile('proguard-android.txt'), 'proguard-rules.pro'
    }
}

系统预置的混淆配置文件proguard-android.txt中配置了-dontpreverify，相当于默认关闭了预校验能力。我们调整一下代码，将系统proguard-android.txt下载下来，移除-dontpreverify，手动关联：

buildTypes {
    release {
        minifyEnabled true
        proguardFiles 'proguard-android-optimize.txt', 'proguard-rules.pro'
    }
    debug {
        minifyEnabled true
        proguardFiles 'proguard-android-optimize.txt', 'proguard-rules.pro'
    }
}

此时我们在proguard-rules.pro同级添加修改后的proguard-android-optimize.txt文件，再次编译Jar文件。发现在Java环境下终于可以执行了。

新问题（待解决）

做了以上配置后，项目依赖使用都没有问题了。但出现了一个新问题：在混淆时，当移除-dontpreverify配置编译了Jar文件，使用《Java代码防止反编译》描述的代码注入方式，防止非法进行反编译会报错，提示：

springboottest/springboot/TestSdkService.java:[46,27] 无法将类 com.sdk.encrypt.sdk中的方法 nSdkDecrypt应用到给定类型;
[ERROR] 需要: java.lang.String,java.lang.String
[ERROR] 找到: java.lang.String
[ERROR] 原因: 实际参数列表和形式参数列表长度不同

初步怀疑时注入代码后，原有的校验被破坏导致的。由于不影响当前功能，所以该问题没有很着急，对应的解决方案还在研究。后面有方案了再整理完善。
